🎨 DESIGN IMPROVEMENTS — EXISTING SCREENS

TYPO: "SETINGS" → "SETTINGS"

Main menu button has a missing T. Fix immediately before any screenshots or demos.

PROFANITY FILTER REQUIRED

Leaderboard contains an offensive username. Implement server-side username validation with a blocklist. Never trust user-submitted display names.

MISSING BUTTON STATES

Design 3 states for every button:

  • Default — normal yellow
  • Pressed — slightly darker, shifted down 2px (pixel push effect)
  • Disabled — gray, non-interactive

INPUT VALIDATION FEEDBACK

Sign In / Sign Up have no error states. Design red border + error text for:

  • Wrong password
  • Username taken
  • Weak password
  • Passwords don't match

VERTICAL SPACING (Login/Settings)

Login and settings screens have cramped vertical rhythm. Add 8–16px more breathing room between elements. Settings particularly feels dense.

CONSOLIDATE SETTINGS SLIDER DESIGN

Settings Page 1 uses +/- circles; Settings Page 2 uses arrow triangles. Pick one. Recommendation: arrow style (more pixel-art native) consistently throughout.

ROUND INDICATOR IN GAMEPLAY

No "Round 1/3" or "Round 2/5" indicator during game. Players can't tell where they are in the match. Add a small round tracker at top center of the game screen.

GENDER SELECTION — ADD "OTHER"

Binary Male/Female only. Add "Other / Prefer not to say" option for inclusivity. This also affects character skin selection scope.

LEADERBOARD — MY RANK ROW

Add a sticky row at the bottom of the leaderboard showing "YOUR RANK: #24" even if you're not in the visible top 8. This is standard UX in competitive mobile games.

ADD DEPTH TO FLAT SCREENS

Some screens (game mode, gender select) feel flat. Add:

  • Subtle cloud parallax movement
  • Small floating particles
  • Light pixel scanline overlay

VICTORY/DEFEAT DIFFERENTIATION

Both screens share very similar layout. Strengthen contrast: Victory should feel warm and bright (more gold, stars, confetti), Defeat should feel distinctly different (darker sky, rain effect, muted colors).

BACK NAVIGATION

Multiple screens missing an explicit back button (Game Mode, Gender Selection). While Android has hardware back, iOS does not. Add visible back arrows to all non-root screens.

🏗️ SYSTEM ARCHITECTURE
Architecture Pattern: Client (React Native) ↔ REST API (Node/Express) + WebSocket (Socket.io) ↔ PostgreSQL + Redis. Firebase handles auth tokens; the backend verifies them. All game logic is server-authoritative to prevent cheating.

Data Models (Core)

// PostgreSQL Schema — Core Tables TABLE users { id UUID PRIMARY KEY username VARCHAR(20) UNIQUE NOT NULL email VARCHAR(255) UNIQUE gender ENUM('male', 'female', 'other') avatar_hat VARCHAR(50) DEFAULT 'default' avatar_shirt VARCHAR(50) DEFAULT 'default' coins INT DEFAULT 100 gems INT DEFAULT 0 rank_points INT DEFAULT 0 win_streak INT DEFAULT 0 created_at TIMESTAMP } TABLE matches { id UUID PRIMARY KEY player1_id UUID REFERENCES users(id) player2_id UUID REFERENCES users(id) -- NULL if vs bot mode ENUM('best_of_3', 'best_of_5') winner_id UUID REFERENCES users(id) score_p1 INT score_p2 INT duration_ms INT created_at TIMESTAMP } TABLE rounds { id UUID PRIMARY KEY match_id UUID REFERENCES matches(id) round_num INT move_p1 ENUM('rock', 'paper', 'scissors') move_p2 ENUM('rock', 'paper', 'scissors') winner ENUM('p1', 'p2', 'draw') } TABLE achievements { id UUID PRIMARY KEY user_id UUID REFERENCES users(id) key VARCHAR(50) -- e.g. 'first_win', 'streak_5' progress INT DEFAULT 0 unlocked_at TIMESTAMP }

WebSocket Events (Multiplayer)

// Client → Server Events join_matchmaking { mode: 'best_of_3' | 'best_of_5' } submit_move { match_id: string, move: 'rock'|'paper'|'scissors' } leave_match { match_id: string } join_lobby { room_code: string } // Server → Client Events match_found { match_id, opponent: { username, avatar, rank } } round_result { round_num, move_p1, move_p2, winner, scores } match_end { winner_id, final_score, xp_gained, coins_gained } opponent_left { reason: 'disconnect' | 'forfeit' } matchmaking_timeout {}

Final Assessment

Your pixel art game already has an exceptionally strong visual identity. The consistency across all 15 screens is genuinely impressive — the color palette, character design, and retro arcade feel are all coherent and polished. Most indie games fail here. You haven't.

The critical gap is the missing system screens (errors, offline, confirmation dialogs) and the progression layer (achievements, profile, daily rewards). These are what transform a fun prototype into an addictive mobile game people return to daily.

With React Native + Expo, your existing designs will translate almost 1:1 to native. The biggest technical challenge will be the real-time multiplayer via WebSockets — invest time getting that architecture right before going to production.

  • Design the 20 missing screens before writing any frontend code
  • Build auth and navigation in Week 1 — it unlocks everything
  • Get bot gameplay working before touching multiplayer
  • Implement the error state system early — don't leave it for last
  • Use EAS Build from day one for easy device testing
  • Set up Sentry before beta — you need crash data from real users
  • Test on a real low-end Android device throughout — don't just use simulators
